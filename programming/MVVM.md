# MVVM 패턴 정리

MVVM은 애플리케이션을 **Model**, **View**, **ViewModel**의 세 가지 역할로 분리하는 아키텍처 패턴입니다.

관심사를 나누어 코드의 **유지보수성**, **테스트 용이성**, **재사용성**을 높이는 것이 핵심입니다.

## 1. MVVM의 3가지 핵심 구성 요소

### Model

애플리케이션의 **실제 데이터**와 **비즈니스 로직**을 담당합니다.

### 주요 역할

- 데이터베이스 접근
- 네트워크 통신(API 호출)
- 데이터 파싱
- 데이터 처리 및 상태 관리

### 핵심 특징

- View나 ViewModel의 존재를 알지 못합니다.
- 오직 데이터와 로직에만 집중합니다.
- UI와 완전히 분리된 독립적인 계층입니다.

---

### View

사용자에게 보여지는 **UI 화면** 자체를 의미합니다.

### 주요 역할

- 화면 렌더링
- 사용자 입력 처리(터치, 클릭 등)
- ViewModel의 상태 변화를 관찰하여 UI 갱신

### 핵심 특징

- 비즈니스 로직을 직접 처리하지 않습니다.
- ViewModel의 데이터를 받아 화면에 반영하는 데 집중합니다.
- 사용자와 시스템 사이의 접점 역할을 합니다.

---

### ViewModel

**View와 Model 사이를 연결하는 중재자** 역할을 합니다.

### 주요 역할

- Model에서 데이터를 받아옴
- View가 사용하기 쉬운 형태로 데이터 가공
- 화면 상태(State) 관리
- 사용자 액션에 따라 필요한 로직 수행

### 핵심 특징

- View에 대한 직접 참조를 가지지 않습니다.
- UI 요소나 화면 제어 코드가 포함되면 안 됩니다.
- View와 느슨하게 결합되어 테스트가 쉽습니다.

---

## 2. MVVM의 동작 흐름

MVVM의 핵심은 **데이터 바인딩(Data Binding)**과 **관찰자 패턴(Observer Pattern)**입니다.

이를 통해 View와 ViewModel은 직접 강하게 결합되지 않고도 자연스럽게 동작합니다.

### 동작 순서

1. 사용자가 **View**에서 액션을 수행합니다.
    
    예: 버튼 클릭, 입력값 변경
    
2. **View**는 해당 액션을 **ViewModel**에 전달합니다.
    
    보통 Command 패턴이나 이벤트 바인딩을 활용합니다.
    
3. **ViewModel**은 필요한 경우 **Model**에 데이터 처리나 갱신을 요청합니다.
4. **Model**은 데이터를 처리한 뒤 결과를 **ViewModel**에 반환합니다.
5. **ViewModel**은 받은 데이터를 가공하여 자신의 **상태(State)**를 업데이트합니다.
6. **View**는 바인딩된 상태 변화를 감지하고 UI를 자동으로 갱신합니다.

---

## 3. MVVM의 장점

### 1) 유닛 테스트가 쉽다

ViewModel이 View에 의존하지 않기 때문에, UI 없이도 로직을 독립적으로 테스트할 수 있습니다.

### 2) 유지보수와 재사용성이 높다

UI와 비즈니스 로직이 분리되어 있어, 화면 디자인이 바뀌더라도 핵심 로직은 그대로 유지할 수 있습니다.

### 3) 역할 분리가 명확하다

개발자는 ViewModel/Model에, 디자이너나 퍼블리셔는 View에 집중할 수 있어 병렬 작업이 수월합니다.

---

## 4. MVVM의 단점

### 1) 학습 곡선이 높다

데이터 바인딩, LiveData, StateFlow, 반응형 프로그래밍 등 관련 개념을 이해해야 하므로 초반 진입 장벽이 있습니다.

### 2) 단순한 화면에는 과할 수 있다

작은 기능에도 계층을 분리하다 보면 클래스와 코드가 불필요하게 늘어나 보일 수 있습니다.

### 3) 상태 관리가 복잡해질 수 있다

프로젝트가 커질수록 ViewModel이 비대해질 수 있고, 바인딩 구조가 복잡해지면 디버깅이 어려워질 수 있습니다.

---

## 5. MVVM 한눈에 보기

| 구성 요소 | 역할 | 특징 |
| --- | --- | --- |
| Model | 데이터 및 비즈니스 로직 처리 | UI와 완전히 분리됨 |
| View | 사용자 인터페이스 표시 | 상태를 관찰하고 화면 갱신 |
| ViewModel | View와 Model 사이 중재 | View를 직접 참조하지 않음 |

---

## 6. 정리

MVVM은 **UI와 로직을 분리하여 구조를 명확하게 만드는 패턴**입니다.

특히 **테스트 용이성**, **유지보수성**, **협업 효율** 측면에서 큰 장점이 있습니다.

다만 모든 상황에 적합한 것은 아니며, 작은 규모의 프로젝트에서는 오히려 구조가 과해질 수 있습니다.

따라서 프로젝트 규모와 복잡도에 맞춰 적절히 도입하는 것이 중요합니다.
