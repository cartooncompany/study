# 웹소켓(WebSocket) 정리

## 한눈에 보는 정의

- **웹소켓(WebSocket)**은

하나의 **TCP 연결** 위에서 **클라이언트와 서버가 서로 자유롭게 데이터를 주고받을 수 있게 해주는 양방향 통신 프로토콜**입니다.

쉽게 말하면,

한 번 연결을 맺은 뒤 계속 열어 둔 상태에서 필요한 순간마다 바로 대화할 수 있는 **직통 통신선**에 가깝습니다.

---

## 어떻게 동작하나요?

### 1. 핸드셰이크(Handshake)

클라이언트가 먼저 일반적인 **HTTP 요청**을 서버로 보냅니다.

이때 요청 헤더에 다음과 같은 의미를 담습니다.

- `Upgrade: websocket`
- “이 연결을 웹소켓 방식으로 바꾸자”

### 2. 연결 전환

서버가 이를 받아들이면, 기존 HTTP 연결은 **웹소켓 연결**로 업그레이드됩니다.

- 비보안: `ws://`
- 보안: `wss://`

### 3. 데이터 교환

연결이 유지되는 동안에는

매번 무거운 HTTP 요청/응답을 반복하지 않고,

가벼운 메시지 프레임으로 데이터를 실시간 전송합니다.

### 4. 연결 종료

클라이언트나 서버 중 한쪽이 종료하기 전까지 연결은 계속 유지됩니다.

---

## HTTP와 웹소켓 비교

| 구분 | HTTP | 웹소켓(WebSocket) |
| --- | --- | --- |
| 통신 방향 | 단방향 요청-응답 | 양방향 통신 |
| 연결 방식 | 요청 후 연결 종료 | 한 번 연결 후 계속 유지 |
| 상태 관리 | 무상태(Stateless) | 상태 유지(Stateful) |
| 오버헤드 | 요청마다 헤더 포함 | 최초 연결 후 오버헤드 적음 |
| 서버 푸시 | 불가능 | 가능 |
| 적합한 용도 | 일반 웹페이지, 게시글 조회/작성 | 실시간 채팅, 시세, 게임, 협업 |

---

## 왜 HTTP보다 유리한가요?

HTTP는 기본적으로

**클라이언트가 요청해야만 서버가 응답**할 수 있습니다.

반면 웹소켓은 연결이 유지되기 때문에

서버도 필요할 때 **클라이언트에게 먼저 데이터 전송(Push)** 이 가능합니다.

즉, 웹소켓은 다음 상황에 특히 강합니다.

- 데이터가 자주 바뀌는 경우
- 반응 속도가 중요한 경우
- 사용자 화면을 실시간으로 업데이트해야 하는 경우

---

## 주요 활용 사례

### 실시간 메신저 / 채팅

예: 카카오톡, 슬랙, 디스코드

- 메시지를 즉시 전달해야 함
- 서버가 새 메시지를 바로 푸시 가능

### 금융 서비스

예: 주식/가상화폐 시세, 호가창

- 가격이 초 단위 이하로 계속 변함
- 빠른 갱신이 핵심

### 멀티플레이어 게임

- 플레이어 위치, 행동, 이벤트를 실시간 동기화
- 지연 시간이 작아야 함

### 협업 툴

예: Google Docs, Figma

- 여러 사용자의 편집 내용을 즉시 반영
- 동시 수정 상태 공유에 적합

### 라이브 서비스

예: 라이브 채팅, 스포츠 스코어보드

- 실시간 반응과 화면 갱신이 중요

---

## 장점

### 1. 빠른 실시간 처리

서버가 클라이언트 요청을 기다리지 않고

필요한 순간 데이터를 바로 보낼 수 있어 **지연 시간이 짧습니다**.

### 2. 네트워크 효율성

Polling처럼 주기적으로 “변경됐나요?”를 묻지 않아도 되므로

**불필요한 요청과 트래픽을 줄일 수 있습니다.**

### 3. 오버헤드 감소

초기 연결 이후에는

반복적인 HTTP 헤더 전송 없이 가벼운 프레임만 주고받아 효율적입니다.

---

## 단점

### 1. 서버 자원 소모

연결을 계속 유지해야 하므로

클라이언트 수가 많아질수록 **메모리와 연결 관리 비용**이 커집니다.

### 2. 확장 구조가 복잡함

서버를 여러 대로 늘릴 때 다음이 까다로워질 수 있습니다.

- 연결 상태 관리
- 세션 유지
- 로드 밸런싱
- 분산 환경에서의 메시지 동기화

즉, **대규모 서비스에서는 인프라 설계 난도가 올라갑니다.**

---

## 언제 쓰면 좋은가요?

웹소켓은 아래 조건에 잘 맞습니다.

- 데이터가 **실시간으로 자주 변경**된다
- 서버가 클라이언트에게 **즉시 알려야** 한다
- 응답 속도가 사용자 경험에 큰 영향을 준다

대표적으로:

- 채팅
- 시세/알림
- 게임
- 협업 편집
- 라이브 대시보드

반대로,

단순 조회/저장 중심의 일반 웹페이지라면

기본 HTTP 방식만으로도 충분한 경우가 많습니다.
