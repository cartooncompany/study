# TCP/IP란 무엇인가

- *TCP/IP(Transmission Control Protocol / Internet Protocol)**는 인터넷과 대부분의 현대 네트워크 통신의 기반이 되는 **프로토콜 집합(protocol suite)**입니다.

즉, 서로 다른 운영체제·하드웨어·네트워크 환경에 있는 장치들이 **같은 규칙으로 데이터를 주고받을 수 있게 해 주는 공통 체계**라고 보면 됩니다. RFC 1122는 이 체계를 호스트 소프트웨어가 따라야 할 통신 계층 요구사항으로 정리하고 있고, TCP는 현재 RFC 9293, IP는 RFC 791을 중심으로 정의됩니다.

---

# 1. TCP와 IP의 역할 분담

TCP/IP라는 이름은 특히 **TCP**와 **IP**라는 두 핵심 프로토콜의 조합을 강조합니다.

둘은 비슷해 보이지만 담당하는 일이 분명히 다릅니다.

## IP: 목적지까지 보내는 규칙

- *IP(Internet Protocol)**는 데이터를 **출발지에서 목적지까지 전달할 수 있게 하는 주소 지정과 패킷 전달 규칙**을 제공합니다.

RFC 791에 따르면 IP는 데이터를 **데이터그램(datagram)** 형태로 전달하며, 출발지와 목적지는 고정 길이 주소로 식별됩니다. 다만 IP 자체는 **데이터가 반드시 도착하는지**, **순서가 보장되는지**, **중간에 유실되지 않았는지**까지 책임지지 않습니다.

정확히 말하면, “IP가 최적 경로를 찾는다”기보다 **라우터들이 자신의 라우팅 정보에 따라 다음 목적지로 전달한다**는 설명이 더 표준 문서에 가깝습니다. ICMP 설명에서도 게이트웨이(라우터)가 라우팅 테이블을 바탕으로 처리함이 드러납니다.

## TCP: 신뢰성 있게 전달하는 규칙

- *TCP(Transmission Control Protocol)**는 IP 위에서 동작하며, 애플리케이션이 주고받는 데이터를 **연결 지향적(connection-oriented)**이고 **신뢰성 있게(reliable)** 전달하도록 설계되었습니다. RFC 9293은 TCP를 인터넷 스택의 핵심 전송 계층 프로토콜로 규정합니다.

TCP는 단순히 데이터를 “쪼개고 번호를 붙이는 것”에 그치지 않습니다. 실제로는 다음을 담당합니다.

- **연결 설정과 종료**
통신 전에 **3-way handshake**로 연결을 맺고, 종료 절차도 따로 관리합니다.
- **순서 보장**
시퀀스 번호와 ACK를 이용해 데이터의 순서를 맞춥니다.
- **재전송**
손실이 감지되면 다시 보냅니다.
- **흐름 제어(flow control)**
수신 측이 감당 가능한 양만 보내도록 윈도우 기반으로 조절합니다.
- **혼잡 제어(congestion control)**
네트워크가 막히지 않도록 전송 속도를 조절합니다. RFC 5681은 slow start, congestion avoidance, fast retransmit, fast recovery를 정의합니다.

# 2. TCP/IP의 계층 구조

실무와 교육에서는 흔히 **TCP/IP 4계층 모델**로 설명합니다. RFC 1122와 RFC 1123도 통신 계층과 응용/지원 프로토콜을 나누어 설명합니다.

## 2.1 애플리케이션 계층 (Application Layer)

사용자가 직접 접하는 프로그램들이 이 계층에서 동작합니다.

웹 브라우저, 메일 클라이언트, 파일 전송 프로그램 등이 여기에 해당합니다. RFC 1123은 응용 및 지원 프로토콜 요구사항을 다룹니다.

대표 프로토콜 예시:

- **HTTP/HTTPS**: 웹
- **SMTP**: 이메일 송신
- **FTP**: 파일 전송
- **DNS**: 이름 해석

## 2.2 전송 계층 (Transport Layer)

애플리케이션이 보낸 데이터를 **프로세스 대 프로세스** 수준으로 전달합니다.

이때 **포트 번호(port number)**를 사용해 어떤 애플리케이션에 전달할지 구분합니다. UDP RFC와 TCP RFC 모두 목적지 포트와 출발지 포트를 정의합니다.

대표 프로토콜:

- **TCP**: 신뢰성, 순서 보장, 연결 지향
- **UDP**: 단순하고 빠르며, 연결 설정 없이 데이터그램 방식으로 전달

## 2.3 인터넷 계층 (Internet Layer)

IP 주소를 기반으로 **호스트 간 데이터그램 전달**을 담당합니다.

이 계층에는 IP 외에도 상태 보고 및 오류 통지를 위한 **ICMP**가 포함됩니다. `ping`이 대표적인 예입니다.

대표 프로토콜:

- **IP**
- **ICMP**

## 2.4 네트워크 접속 계층 (Network Access / Link Layer)

실제 물리 네트워크에 데이터를 실어 보내는 계층입니다.

이더넷, Wi-Fi 같은 기술과 연결되며, 프레임 단위 전송과 로컬 링크 수준의 주소 체계를 다룹니다. RFC 1122도 링크 계층을 통신 계층 구조의 일부로 다룹니다.

# 3. 데이터는 어떻게 이동하는가: 캡슐화

TCP/IP 통신은 흔히 **캡슐화(encapsulation)**로 설명합니다.

애플리케이션이 만든 데이터에, 아래 계층으로 내려가면서 각 계층이 필요한 제어 정보를 덧붙입니다. 반대로 수신 측에서는 이를 벗겨내며 원래 데이터를 복원합니다. 이 구조는 TCP/IP 계층 모델의 핵심 동작 방식입니다.

예를 들어 웹 페이지 요청이 전송될 때는 대략 이런 흐름입니다.

1. 애플리케이션 계층에서 HTTP 요청 생성
2. 전송 계층에서 TCP 헤더 추가
3. 인터넷 계층에서 IP 헤더 추가
4. 링크 계층에서 프레임 헤더 추가
5. 실제 신호로 전송

수신 측은 이 과정을 거꾸로 수행합니다.

즉, **프레임 → IP 패킷(정확히는 IP 데이터그램) → TCP 세그먼트 → 애플리케이션 데이터** 순서로 해석합니다. TCP와 IP RFC는 각각 세그먼트와 데이터그램의 형식을 정의합니다.

# 4. 자주 헷갈리는 핵심 개념

## 4.1 패킷, 세그먼트, 데이터그램은 완전히 같은 말이 아니다

입문 자료에서는 전부 “패킷”이라고 뭉뚱그려 부르지만, 엄밀히는 계층마다 표현이 다릅니다.

- **애플리케이션 데이터**: 프로그램이 만든 원본 데이터
- **TCP 세그먼트(segment)**: TCP 헤더가 붙은 전송 단위
- **IP 데이터그램(datagram)**: IP 헤더가 붙은 인터넷 계층 단위
- **링크 프레임(frame)**: 실제 네트워크 매체를 타고 가는 단위

즉, “TCP가 패킷으로 쪼갠다”는 표현은 쉬운 설명으로는 괜찮지만, 기술적으로는 **TCP 세그먼트로 나눈다**가 더 정확합니다.

## 4.2 포트 번호는 “어느 컴퓨터인가”가 아니라 “그 컴퓨터의 어느 프로그램인가”를 가리킨다

IP 주소가 **장치(호스트)**를 식별한다면, 포트 번호는 그 장치 안의 **특정 애플리케이션 프로세스**를 식별합니다. TCP와 UDP 모두 헤더에 출발지/목적지 포트를 포함합니다.

예:

- IP 주소: 어느 집인지
- 포트 번호: 그 집의 어느 방인지

## 4.3 TCP는 “정확성”, UDP는 “속도”라고만 외우면 부족하다

이 설명은 방향성은 맞지만 너무 단순합니다.

UDP는 TCP보다 단순하고 연결 설정이 없어서 지연이 적을 수 있지만, “항상 더 빠르다”기보다 **신뢰성·순서 보장·재전송이 없는 대신 오버헤드가 적다**가 정확한 설명입니다. 따라서 실시간 음성/영상, 게임, DNS처럼 **약간의 손실보다 지연이 더 치명적인 상황**에서 자주 쓰입니다. 반면 파일 전송, 웹 문서 전송, 로그인 세션처럼 **정확성이 중요한 경우**에는 TCP가 적합합니다.

# 5. TCP에서 꼭 추가해야 하는 개념

Gemini의 원문에서 가장 아쉬운 부분은, TCP의 핵심을 “재조립과 재전송” 정도로만 설명한 점입니다. 실제로는 아래 개념이 아주 중요합니다.

## 5.1 3-way handshake

TCP는 통신 전에 보통 다음 절차로 연결을 맺습니다.

1. 클라이언트가 **SYN**
2. 서버가 **SYN-ACK**
3. 클라이언트가 **ACK**

RFC 9293은 이를 TCP 연결 설정 절차로 명시합니다.

이 절차의 목적은 단순 인사가 아니라,

- 양쪽이 통신 가능한지 확인하고
- 초기 시퀀스 번호를 합의하며
- 연결 상태를 서로 맞추는 데 있습니다.

## 5.2 흐름 제어 (Flow Control)

수신 측이 너무 느리면 송신 측이 마구 보내면 안 됩니다.

TCP는 **수신 윈도우(receive window)** 개념을 통해 상대가 받을 수 있는 양만큼만 보내도록 조절합니다.

## 5.3 혼잡 제어 (Congestion Control)

문제가 수신 측이 아니라 **네트워크 전체**에 있을 수도 있습니다.

RFC 5681은 TCP가 네트워크 혼잡을 피하기 위해 **slow start**, **congestion avoidance**, **fast retransmit**, **fast recovery** 같은 알고리즘을 사용한다고 설명합니다.

즉,

- **흐름 제어**는 “상대방이 받을 수 있나?”
- **혼잡 제어**는 “네트워크가 버틸 수 있나?”

를 다루는 서로 다른 개념입니다.

# 6. ICMP는 왜 중요한가

입문 설명에서 ICMP는 종종 한 줄로 끝나지만, 실제 네트워크 이해에는 꽤 중요합니다.

ICMP는 IP 계층과 함께 동작하며, **오류 보고와 진단 메시지 전달**에 쓰입니다. RFC 792는 라우팅 과정에서 문제가 생기거나 TTL이 만료되는 등의 상황을 ICMP 메시지로 알리는 구조를 정의합니다.

대표 예시:

- **ping**: 상대가 응답 가능한지 확인
- **Destination Unreachable**: 목적지에 갈 수 없음
- **Time Exceeded**: TTL 만료

즉, ICMP는 데이터를 실어 나르는 주인공이라기보다, **네트워크 상태를 알려주는 제어·진단 도구**에 가깝습니다.

# 7. TCP/IP와 OSI 7계층은 어떻게 다른가

많은 설명이 TCP/IP와 OSI를 섞어 말하는데, 이 둘은 목적이 조금 다릅니다.

- **TCP/IP 모델**: 실제 인터넷 프로토콜 스택을 설명하는 데 더 가깝습니다. RFC 1122/1123은 이 관점에서 계층 요구사항을 다룹니다.
- **OSI 7계층**: 교육과 개념 정리에 유용한 참조 모델입니다.

대략적으로 대응시키면:

- TCP/IP **애플리케이션 계층** ↔ OSI **응용/표현/세션**
- TCP/IP **전송 계층** ↔ OSI **전송**
- TCP/IP **인터넷 계층** ↔ OSI **네트워크**
- TCP/IP **네트워크 접속 계층** ↔ OSI **데이터 링크/물리**

즉, “TCP/IP는 4계층, OSI는 7계층”이라고 외우는 것보다,

**TCP/IP는 실제 인터넷 구현 중심**, **OSI는 개념 학습 중심**이라고 이해하는 편이 더 실용적입니다. 이 비교는 RFC 구조 자체에서도 TCP/IP가 실제 프로토콜 묶음으로 다뤄진다는 점에서 뒷받침됩니다.

# 8. 한 번에 이해하는 예시: 웹 페이지를 열 때

브라우저로 웹사이트를 여는 과정을 TCP/IP 관점에서 단순화하면 이렇습니다.

## 1) 애플리케이션 계층

브라우저가 HTTP 요청을 만듭니다.

필요하면 먼저 DNS를 통해 도메인 이름을 IP 주소로 바꿉니다. RFC 1123은 이런 응용/지원 프로토콜 요구사항을 다룹니다.

## 2) 전송 계층

브라우저는 보통 TCP를 사용해 서버와 연결을 맺습니다.

연결 후 HTTP 요청 데이터는 TCP 세그먼트로 전송됩니다.

## 3) 인터넷 계층

각 TCP 세그먼트는 IP 데이터그램에 실립니다.

이후 여러 라우터를 거쳐 목적지 서버로 전달됩니다.

## 4) 링크 계층

각 구간에서 이더넷이나 Wi-Fi 프레임으로 실려 실제 전기·무선 신호로 이동합니다.

## 5) 수신 측

서버는 헤더를 순서대로 해석해 HTTP 요청을 복원하고, 다시 응답을 같은 방식으로 돌려보냅니다.

# 9. 원문 설명에서 보완하면 더 좋아지는 부분

Gemini 답변은 입문 설명으로는 좋지만, 아래처럼 다듬으면 더 정확합니다.

## 보완 포인트 1

기존 표현:

**“IP가 최적의 경로를 찾는다”**

더 정확한 표현:

**IP는 주소 지정과 데이터그램 전달의 규칙을 제공하고, 실제 경로 선택은 라우터들이 라우팅 정보에 따라 수행한다.**

## 보완 포인트 2

기존 표현:

**“TCP는 데이터를 패킷으로 쪼개고 순서를 맞춘다”**

더 정확한 표현:

**TCP는 연결 설정, 순서 보장, 재전송, 흐름 제어, 혼잡 제어를 통해 신뢰성 있는 바이트 스트림 전달을 제공한다.**

## 보완 포인트 3

기존 표현:

**“UDP는 속도 우선”**

더 정확한 표현:

**UDP는 연결 설정과 신뢰성 제어가 없는 단순한 데이터그램 전송 프로토콜이며, 낮은 지연과 적은 오버헤드가 필요한 상황에 적합하다.**

# 10. 한눈에 보는 정리

## TCP/IP의 본질

- 인터넷 통신을 위한 **표준 프로토콜 집합**
- 서로 다른 시스템끼리도 통신 가능하게 만드는 공통 규칙

## IP의 역할

- 주소 지정
- 데이터그램 전달
- 종단 간 신뢰성은 보장하지 않음

## TCP의 역할

- 연결 설정
- 순서 보장
- 재전송
- 흐름 제어
- 혼잡 제어

## UDP의 역할

- 단순한 데이터그램 전달
- 연결 설정 없음
- 지연 민감한 통신에 적합

## TCP/IP 4계층

- 애플리케이션
- 전송
- 인터넷
- 네트워크 접속
