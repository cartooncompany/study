# RDBMS vs NoSQL

데이터베이스를 공부할 때 가장 먼저 자주 비교하게 되는 개념이 **RDBMS(관계형 데이터베이스)** 와 **NoSQL(비관계형 데이터베이스)** 입니다.

둘 중 무엇이 더 우월하다고 보기보다는, **서비스의 요구사항에 따라 적합한 선택이 달라진다**고 이해하는 것이 중요합니다.

---

## 1. 한눈에 보는 RDBMS vs NoSQL 비교표

| 비교 항목 | RDBMS (관계형 DB) | NoSQL (비관계형 DB) |
| --- | --- | --- |
| **데이터 구조** | 테이블(Table), 행(Row), 열(Column) 기반 | 문서(Document), 키-값(Key-Value), 컬럼 패밀리, 그래프 등 다양한 형태 |
| **스키마** | 고정된 스키마, 사전 설계 필요 | 유연한 스키마, 동적 변경 가능 |
| **데이터 관계** | 테이블 간 관계를 명확하게 정의 가능 | 관계 표현이 제한적이거나 애플리케이션 레벨에서 처리 |
| **확장 방식** | 주로 수직 확장(Scale-up) | 주로 수평 확장(Scale-out) |
| **트랜잭션** | ACID 보장에 강함 | BASE 기반이 많고, 최종 일관성을 택하는 경우가 많음 |
| **일관성** | 강한 일관성(Strong Consistency) | 유연한 일관성(Eventual Consistency) 중심 |
| **쿼리 방식** | SQL 사용, 복잡한 조회와 조인에 강함 | DB별 API/전용 쿼리 방식 사용 |
| **조인(Join)** | 매우 강력함 | 일반적으로 제한적이거나 비효율적 |
| **대표 제품** | MySQL, PostgreSQL, Oracle, SQL Server | MongoDB, Redis, Cassandra, DynamoDB, Neo4j |
| **적합한 환경** | 정확성, 무결성, 복잡한 조회가 중요한 서비스 | 대규모 분산 처리, 유연한 구조, 빠른 확장이 중요한 서비스 |

---

## 2. RDBMS란?

RDBMS는 **Relational Database Management System**의 약자로, 데이터를 **테이블 형태**로 저장하고, 각 테이블 사이의 **관계(Relation)** 를 정의해 관리하는 데이터베이스입니다.

예를 들어 다음처럼 데이터를 나눠 관리할 수 있습니다.

- `users` 테이블: 사용자 정보 저장
- `orders` 테이블: 주문 정보 저장
- `products` 테이블: 상품 정보 저장

이때 `orders` 테이블은 `users`, `products` 와 연결되며, 이런 관계를 통해 데이터 중복을 줄이고 체계적으로 관리할 수 있습니다.

---

## 3. RDBMS의 특징

### 3.1 장점

### 1) 명확한 데이터 구조

테이블과 컬럼이 미리 정의되어 있어 데이터 구조를 이해하기 쉽습니다.

협업하거나 유지보수할 때도 구조가 비교적 명확합니다.

### 2) 데이터 무결성과 일관성 보장

RDBMS는 보통 **ACID** 특성을 충실히 지원합니다.

- **Atomicity(원자성)**: 작업이 모두 성공하거나 모두 실패해야 함
- **Consistency(일관성)**: 데이터가 항상 유효한 상태를 유지
- **Isolation(고립성)**: 동시에 실행되는 트랜잭션이 서로 간섭하지 않음
- **Durability(지속성)**: 한 번 완료된 작업은 장애가 발생해도 유지됨

이 특성 덕분에 **결제, 금융, 재고 관리**처럼 오류가 치명적인 분야에 적합합니다.

### 3) 복잡한 조회에 강함

SQL을 사용하면 여러 테이블을 `JOIN` 해서 복잡한 조건의 데이터를 쉽게 조회할 수 있습니다.

예:

- 특정 기간 동안 주문한 사용자 목록
- 가장 많이 팔린 상품 순위
- 사용자별 총 결제 금액 집계

### 4) 표준화된 언어 사용

대부분의 관계형 DB는 SQL을 사용하므로, 특정 제품에 종속되지 않고 학습한 내용을 다른 DB에도 어느 정도 적용할 수 있습니다.

---

### 3.2 단점

### 1) 스키마 변경이 까다로움

초기 설계를 잘못하면 나중에 컬럼 추가, 타입 변경, 관계 변경이 번거롭고 비용이 큽니다.

### 2) 대규모 수평 확장이 상대적으로 어려움

데이터가 폭발적으로 증가하는 환경에서는 서버 여러 대에 나누는 구조가 복잡해질 수 있습니다.

그래서 주로 더 높은 성능의 서버로 업그레이드하는 **수직 확장**에 의존하는 경우가 많습니다.

### 3) 비정형 데이터 처리에 불리함

모든 데이터가 동일한 구조를 가지지 않는 경우, 테이블 구조로 표현하는 것이 오히려 불편할 수 있습니다.

예:

- 사용자마다 서로 다른 프로필 정보
- 자유로운 JSON 형태의 메타데이터
- 빠르게 바뀌는 요구사항

---

## 4. RDBMS가 잘 맞는 경우

다음과 같은 프로젝트에는 RDBMS가 적합합니다.

- **금융 시스템**
- **결제 시스템**
- **ERP / 회계 시스템**
- **회원, 주문, 재고 관리**
- **복잡한 검색과 집계가 많은 서비스**
- **데이터 정확성과 신뢰성이 가장 중요한 서비스**

핵심은 다음과 같습니다.

> 구조가 명확하고, 데이터 오류가 절대 발생하면 안 되며, 관계형 조회가 많은 경우
> 
> 
> → **RDBMS가 유리하다**
> 

---

## 5. NoSQL이란?

NoSQL은 **Not Only SQL**의 약자로, 관계형 데이터베이스처럼 반드시 테이블 구조를 따르지 않는 데이터베이스를 의미합니다.

즉, “SQL을 쓰지 않는다”라기보다 **관계형 모델에만 제한되지 않는 다양한 저장 방식**을 포함하는 개념입니다.

NoSQL은 보통 다음과 같은 형태로 구분됩니다.

- **Key-Value Store**: Redis
- **Document Store**: MongoDB
- **Column Family Store**: Cassandra
- **Graph Database**: Neo4j

---

## 6. NoSQL의 특징

### 6.1 장점

### 1) 뛰어난 유연성

NoSQL은 스키마가 엄격하지 않은 경우가 많아서, 데이터 구조를 유연하게 바꿀 수 있습니다.

예를 들어 사용자 정보에서:

- 어떤 사용자는 `phone` 필드가 있고
- 어떤 사용자는 `address` 필드가 있고
- 어떤 사용자는 둘 다 없어도 저장 가능

이런 특성은 요구사항이 자주 바뀌는 서비스에 매우 유리합니다.

### 2) 수평 확장이 쉬움

NoSQL은 처음부터 분산 환경을 고려해 설계된 경우가 많아, 서버 여러 대에 데이터를 분산 저장하는 데 유리합니다.

즉:

- 데이터가 많아질수록
- 서버를 여러 대 추가해
- 성능과 저장 용량을 늘리기 쉬움

대규모 트래픽 서비스에 적합한 이유입니다.

### 3) 빠른 읽기/쓰기 처리

복잡한 조인 없이 필요한 데이터를 한 번에 가져오도록 설계하는 경우가 많아서, 읽기/쓰기 성능에 유리할 수 있습니다.

### 4) 비정형/반정형 데이터에 적합

로그, 이벤트 데이터, 사용자 활동 기록, JSON 문서 같은 구조가 자주 바뀌는 데이터 저장에 강합니다.

---

### 6.2 단점

### 1) 일관성이 약해질 수 있음

분산 시스템에서는 성능과 가용성을 높이기 위해 **강한 일관성보다 최종 일관성(Eventual Consistency)** 을 선택하는 경우가 많습니다.

즉, 어떤 시점에는 여러 서버의 데이터가 잠깐 다를 수 있습니다.

### 2) 복잡한 조회에 불리할 수 있음

RDBMS처럼 표준 SQL과 강력한 조인을 제공하지 않는 경우가 많아서, 복잡한 관계형 분석에는 불편할 수 있습니다.

### 3) 데이터 중복이 늘어날 수 있음

조인을 덜 사용하기 위해 관련 데이터를 한 문서 안에 함께 저장하는 경우가 많습니다.

이렇게 하면 조회는 빨라질 수 있지만, 중복과 수정 비용이 커질 수 있습니다.

### 4) 제품별 학습 방식이 다름

NoSQL은 종류가 매우 다양해 DB마다 사용법, 쿼리 방식, 설계 철학이 다릅니다.

즉, 하나를 배웠다고 해서 다른 NoSQL에도 바로 적용되지는 않습니다.

---

## 7. NoSQL이 잘 맞는 경우

다음과 같은 프로젝트에는 NoSQL이 적합합니다.

- **대규모 로그 수집 시스템**
- **실시간 채팅 / 알림 시스템**
- **SNS 피드 / 콘텐츠 서비스**
- **IoT 센서 데이터 저장**
- **빅데이터 처리**
- **사용자별로 구조가 자주 바뀌는 프로필 데이터**
- **짧은 응답 시간과 빠른 확장이 중요한 서비스**

핵심은 다음과 같습니다.

> 데이터 구조가 자주 변하고, 데이터 양이 매우 많으며, 빠른 확장성과 유연성이 중요한 경우
> 
> 
> → **NoSQL이 유리하다**
> 

---

## 8. ACID vs BASE

RDBMS와 NoSQL을 비교할 때 자주 등장하는 개념입니다.

### ACID

RDBMS가 중요하게 여기는 특성으로, **정확성과 신뢰성**에 초점이 있습니다.

- **Atomicity**: 전부 성공 또는 전부 실패
- **Consistency**: 항상 일관된 상태 유지
- **Isolation**: 동시에 처리되어도 충돌 최소화
- **Durability**: 완료된 데이터는 보존

### BASE

NoSQL 계열에서 자주 언급되는 개념으로, **가용성과 확장성**에 초점이 있습니다.

- **Basically Available**: 시스템은 항상 응답 가능해야 함
- **Soft State**: 상태가 일시적으로 일관되지 않을 수 있음
- **Eventually Consistent**: 시간이 지나면 최종적으로 일관된 상태가 됨

정리하면:

- **ACID = 정확성 중심**
- **BASE = 확장성과 가용성 중심**

다만 최근에는 NoSQL도 트랜잭션을 일부 지원하고, RDBMS도 분산 확장을 지원하는 등 경계가 예전보다 조금씩 흐려지고 있습니다.

---

## 9. RDBMS와 NoSQL의 설계 철학 차이

두 데이터베이스는 단순히 저장 방식만 다른 것이 아니라, **데이터를 바라보는 관점 자체가 다릅니다.**

### RDBMS의 관점

- 데이터 중복을 최소화
- 관계를 명확히 정의
- 정규화(Normalization)를 통해 체계적으로 관리
- 무결성과 정확성을 최우선으로 함

### NoSQL의 관점

- 조회 성능과 확장성 우선
- 필요한 데이터를 한 번에 읽기 쉽게 구성
- 비정규화(Denormalization)를 적극 활용
- 분산 환경에서 유연하게 운영 가능하도록 설계

즉:

- **RDBMS는 데이터 구조의 정합성**
- **NoSQL은 서비스 운영의 유연성과 성능**

에 더 초점을 맞춘다고 볼 수 있습니다.

---

## 10. 실제 서비스에서는 어떻게 사용할까?

현실의 대규모 서비스는 하나의 DB만 고집하지 않는 경우가 많습니다.

이를 **Polyglot Persistence**라고 합니다.

예를 들면:

- **회원, 주문, 결제 정보** → RDBMS
- **캐시 데이터** → Redis
- **로그/이벤트 데이터** → NoSQL
- **추천 그래프/연결 관계 분석** → Graph DB

즉, 데이터의 성격에 따라 다른 저장소를 함께 사용합니다.

### 예시

- **전자상거래 서비스**
    - 주문/결제: RDBMS
    - 상품 조회 캐시: Redis
    - 사용자 행동 로그: MongoDB 또는 Cassandra
- **SNS 서비스**
    - 회원 정보: RDBMS
    - 피드/게시물 메타데이터: Document DB
    - 실시간 캐시/세션: Redis

---

## 11. 선택 기준 정리

### RDBMS를 선택하면 좋은 경우

- 데이터 구조가 명확함
- 데이터 무결성이 매우 중요함
- 복잡한 조회와 조인이 많음
- 트랜잭션 보장이 필요함

### NoSQL을 선택하면 좋은 경우

- 데이터 구조가 자주 바뀜
- 빠른 확장성이 필요함
- 대량의 비정형 데이터를 다룸
- 복잡한 조인보다 빠른 읽기/쓰기가 중요함

---

## 12. 최종 요약

RDBMS와 NoSQL은 경쟁 관계라기보다, **서로 다른 문제를 해결하기 위한 도구**입니다.

- **RDBMS**는
데이터의 **정확성**, **무결성**, **복잡한 관계 처리**에 강합니다.
- **NoSQL**은
데이터의 **유연성**, **확장성**, **대규모 분산 처리**에 강합니다.

따라서 중요한 것은

**“어떤 DB가 더 좋은가?”** 가 아니라

**“우리 서비스에 어떤 특성이 더 중요한가?”** 입니다.

---

## 13. 한 문장으로 정리

> **정확성과 관계 중심이면 RDBMS, 유연성과 확장성 중심이면 NoSQL을 우선 고려한다.**
>
